# Prop AMM Review Findings

## Findings

1. High: `open_swap` sell creates a second redemption path that bypasses the existing redemption control plane and fee model. In [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:503), sell goes straight from `process_redemption_core` into `execute_redemption_operations` and pays out from the redemption vault, without requiring a `RedemptionOffer`, `redemption_admin`, or the existing redemption fee routing checks enforced by [fulfill_redemption_request.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/redemption/fulfill_redemption_request.rs:223). It also hardcodes zero redemption fee in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:590). If the queued/admin redemption path is meant to be authoritative, this is a real business-logic and authorization bypass.

2. High: sell-side accounting diverges from the canonical redemption flow. `execute_open_swap_sell` burns/transfers ONYC and refreshes market stats, but it does not run the buffer accrual/post-supply handling that the existing redemption path performs in [fulfill_redemption_request.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/redemption/fulfill_redemption_request.rs:522). On top of that, it refreshes market stats using `ctx.accounts.token_in_mint` without reloading the mint after burn in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:613), unlike the canonical redemption flow which reloads before recomputing stats in [fulfill_redemption_request.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/redemption/fulfill_redemption_request.rs:584). That can leave circulating supply/TVL stale or inconsistent after sell swaps.

3. High: the new AMM surface is not actually isolated; the buy path is duplicated and tightly coupled to legacy offer logic, increasing drift risk. `prop_amm/swap.rs` directly embeds approval checks, ATA handling, buffer accrual, token execution, and market-stats refresh for buy in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:324), while `take_offer.rs` now has similar quote/min-out handling in [take_offer.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/offer/take_offer.rs:428). Any future fix to approvals, account validation, accrual order, or stats refresh now has to be kept in sync across both modules.

4. High: `quote_swap` can advertise liquidity that is not actually executable. It ignores approval requirements and transfer-fee incompatibilities when building the quote in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:204), but `open_swap` later enforces approvals in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:349) and [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:518), and execution-time token logic rejects transfer-fee mints downstream. From Titan/Jupiter’s perspective, the venue can return a valid-looking quote that is guaranteed to fail on swap.

5. High: the swap interface imposes too much venue-internal account burden on integrators. `OpenSwap` always requires offer-vault accounts, redemption-vault accounts, buffer accounts, `market_stats`, `main_offer`, and `offer_vault_onyc_account` in one context in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:67), even though buy and sell use different subsets. The TS wrapper in [onre_program.ts](/Users/marian-danielrolnik/Documents/projects/onre-sol/tests/onre_program.ts:327) shows the resulting account fanout. That is likely integration friction for Titan because the adapter has to learn your full internal topology, not just canonical market accounts for the chosen side.

6. Medium: users can be forced to pay rent to initialize protocol-owned accounts on first sell. In the sell path, `boss_token_in_account` and `redemption_vault_token_in_account` are created with `payer: user` in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:548) and [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:561). That is poor UX for an aggregator route and it is not surfaced in the quote.

7. Medium: the interface is still effectively offer-centric and the canonical offer derivation rule is duplicated across the program and both client helpers. The program validates `(asset, ONYC)` in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:188), but the same rule is reimplemented in [builders_offer.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/tests/common/builders_offer.rs:409) and [onre_program.ts](/Users/marian-danielrolnik/Documents/projects/onre-sol/tests/onre_program.ts:291). If that canonical routing rule changes later, both tests and external adapters break.

8. Medium: sell-side zero-amount swaps appear to be accepted. `process_offer_core` now rejects zero amount in [offer_utils.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/offer/offer_utils.rs:96), but the sell path relies on `process_redemption_core`, which does not have the same guard, and `execute_open_swap_sell` does not add one before doing ATA creation and event emission in [swap.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/src/instructions/prop_amm/swap.rs:535). That is mostly spam/surface area, but it should be closed.

9. Medium: test coverage does not hit the riskiest new behavior. The dedicated AMM suite in [prop_amm.rs](/Users/marian-danielrolnik/Documents/projects/onre-sol/programs/onreapp/tests/prop_amm.rs:57) covers happy-path quote/buy/sell and expiry/min-out, but it does not cover approval-gated offers, transfer-fee mint incompatibility, sell-side market stats after burn, or buffer-managed ONYC behavior.

## Open Question

If the intent is explicitly to create a privileged “instant redemption AMM” that bypasses the queued/admin redemption flow, then finding 1 is a deliberate product change rather than a bug. If that is the intended design, it still needs to be made explicit in docs, fees, and accounting, because right now it silently diverges from the rest of the redemption system.
